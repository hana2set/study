MQ 목적
- 비동기 처리 구현이 어려움
  - Java NIO를 통해 구현해야 효율적임
  - 데이터 유실 및 작업 실패에 대한 방어책을 구성해야함
- 서버 메모리의 휘발성 극복 (아키텍처의 한계)
  - 데이터를 "메모리"에 등록해서 처리해야함.
  - 작업 실패 시 복구가 어려움
  - **MQ는 메세지를 받자마자 하드디스크에 저장**하고 시작.
- 시스템 간 결합도 낮춤


MQ가 메세지 인식하는 방식
- Pull 방식 (Kafka)
  - loop 사용
  - Java NIO의 selector 기술 사용.
    - OS의 네트워크 감시 기능에게 소켓에 들어오는 데이터 감시 요청 (epoll, kqueue, window-IOCP)
    - 그동안 자바 스레드는 대기상태 -> CPU 사용 안함
- Push 방식 (RabbitQ)
  - Customer가 리스너 함수를 등록하고, Broker가 직접 push
